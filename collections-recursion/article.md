Как-то раз на одном из программистских форумов я наткнулся на интересную задачку. Интересна она была тем, что
требовалось рекурсивное решение на Java. Мне захотелось разобраться...

## Задача

Написать обобщённый метод, который принимает в качестве аргумента коллекцию объектов и модифицирует её следующим
образом:

1. Если в коллекции встречается только один объект, оставить его как есть.
2. Если в коллекции встречаются подряд два одинаковых объекта и более, удалить их все и поместить на это место `null`.
3. Метод должен быть рекурсивным.

Примеры:

* `[1, 2, 2]` → `[1, null]`
* `["dog", "cat", "cat", "fish"]` → `["dog", null, "fish"]`
* `[1, 2, 1, 3, 1]` → `[1, 2, 1, 3, 1]`

## Чёткое условие - это половина решения

Очевидно, что настоящий метод (то есть предназначенный для вызова на объекте) нам не понадобится. Хватит статического.
По сути это никакой не метод, а самая обычная процедура. Или нет? Сложно сказать...

Во-первых, коллекцию можно обойти только с помощью итератора (имеется ввиду интерфейс `java.util.Collection`, а не
конкретная реализация, например, `java.util.ArrayList`). Причем единственная возможная модификация коллекции через
итератор - удаление элемента. Причём если один итератор задействован для обхода коллекции, а другой - для удаления, даже
попеременное выполнение этих операций в одном потоке недопустимо.

```java
public class ConcurrentModificationExample {
    public static void main(String[] args) {
        Set<Integer> hashSet = new HashSet<>();
        hashSet.add(41);
        hashSet.add(19);
        hashSet.add(43);
        hashSet.add(13);

        var iterator1 = hashSet.iterator();
        var iterator2 = hashSet.iterator();

        iterator1.next();
        iterator2.next();
        iterator2.remove();
        iterator1.next();
    }
}
```

Запустим код из листинга 1 и увидим ошибку:

```
Exception in thread "main" java.util.ConcurrentModificationException
	at java.base/java.util.HashMap$HashIterator.nextNode(HashMap.java:1606)
	at java.base/java.util.HashMap$KeyIterator.next(HashMap.java:1629)
	at ConcurrentModificationExample.main(ConcurrentModificationExample.java:19)
```

Допустим, мы каким-то магическим образом обошли это ограничение (на самом деле, способ есть, но он не решает
принципиальной проблемы, о которой чуть позже). Мы удалили из исходной коллекции несколько одинаковых идущих подряд
элементов. Как мы добавим в коллекцию наш заместитель, то есть `null`? Итераторы такой возможности не предоставляют. 

Тут вполне резонно возражение, как же так? Ведь в интерфейсе `Collection` есть метод `add()`. Вот только непонятно, 
куда добавится элемент. В случае списка, в конец. В случае множества - зависит от реализации. Так, в случае 
`LinkedHashSet` элементы добавляются в конец. А в случае с `HashSet` неизвестно куда. Да толку нам от 
гарантированного добавления в конец, если нам нужно заменить элемент(ы) в середине коллекции.

Ну и самое главное. По отношению к множествам вся эта задача не имеет никакого смысла. Так как во множестве хранится по 
одному элементу на каждое значение. Впрочем, нам это не мешает. Множество, так множество - это вырожденный, но 
допустимый случай.

Важно вот что: править коллекцию "на лету" нельзя. Вместо процедуры будет функция, которая складывает обработанные 
элементы в новую коллекцию. Ох, полегчало.
