# Неочевидные преимущества непопулярных коллекций

Когда встаёт выбор, какую реализацию коллекции задействовать, то в четырёх случаях из пяти берётся что-то со словом
hash, например, HashMap. Реже используется TreeMap. А LinkedHashMap не пользуется полулярностью. Зато на
собеседовании обязательно спросят, что же такого в ней особенного. А отвечать принято про всякие там 
добавления, итерации и прочие преждевременные оптимизации. 

В этой статье различные реализации коллекций рассматриваются под другим углом: удобство отладки. Мы рассмотрим 
только штатные реализации интерфейса Set. С Map похожая история. А со списками и так всё понятно. 

Начнём, пожалуй.

## Set

Вглянем на листинг 1. Понятно, что инициализировать коллекцию таким образом в наши времена не принято. Но это не 
инициализация, это имитация заполнения коллекции где-то в дебрях кода.

```java
import java.util.HashSet;
import java.util.Set;

public class Listing1 {
    public static void main(String[] args) {
        Set<String> hashSet = new HashSet<>();
        hashSet.add("Апельсин");
        hashSet.add("Банан");
        hashSet.add("Лимон");

        System.out.println(hashSet);
    }
}
```

Получим следующий вывод:

```
[Лимон, Апельсин, Банан]
```

Ничего необычного: порядок следования элементов в коллекции не сохраняется.

Если добавить абрикос, то бишь после добавления лимона сделать `hashSet.add("Абрикос")`, то получим 
`[Лимон, Абрикос, Апельсин, Банан]`. Вроде бы снова ничего необычного.
